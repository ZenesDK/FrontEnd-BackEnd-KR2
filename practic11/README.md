# Практическая работа №11: RBAC (Role-Based Access Control)

## 📌 Описание работы

В данной практической работе реализована система управления доступом на основе ролей (RBAC) для интернет-магазина. Проект демонстрирует полный цикл разработки веб-приложения: от аутентификации до разграничения прав доступа.

---

## 🎯 Цель работы

Реализовать систему ролей с разграничением прав доступа:
- **Гость** (неавторизованный пользователь) — только регистрация и вход
- **Пользователь** — просмотр товаров
- **Продавец** — управление товарами (создание, редактирование)
- **Администратор** — полный доступ (управление пользователями + товарами)

---

## 🛠 Используемые технологии и их назначение

### 🔹 Бэкенд (серверная часть)

| Технология | Для чего используется | Где реализовано |
|------------|----------------------|-----------------|
| **Node.js** | Среда выполнения JavaScript на сервере | `backend/server.js` |
| **Express** | Фреймворк для создания REST API, обработка маршрутов | `backend/server.js` — все маршруты |
| **JWT (jsonwebtoken)** | Создание и проверка access/refresh токенов для аутентификации | `generateAccessToken()`, `generateRefreshToken()`, `authMiddleware` |
| **bcrypt** | Хеширование паролей пользователей (безопасное хранение) | `hashPassword()`, `verifyPassword()` |
| **nanoid** | Генерация уникальных идентификаторов для пользователей и товаров | `id: nanoid(6)` при создании |
| **Swagger (swagger-jsdoc, swagger-ui-express)** | Автоматическая генерация интерактивной документации API | `http://localhost:3000/api-docs` |
| **CORS** | Разрешение кросс-доменных запросов между фронтендом (3001) и бэкендом (3000) | `app.use(cors({ origin: "http://localhost:3001" }))` |
| **JSON файл** | Хранение данных (пользователи, товары, токены)| `db.json` — все данные сохраняются между перезапусками |

### 🔸 Фронтенд (клиентская часть)

| Технология | Для чего используется | Где реализовано |
|------------|----------------------|-----------------|
| **React** | Библиотека для построения пользовательского интерфейса | Все файлы в `frontend/src/` |
| **React Router v6** | Маршрутизация между страницами (логин, регистрация, товары, пользователи) | `App.js` — компонент `Routes` |
| **Axios** | HTTP-клиент для отправки запросов к API | `api/index.js` — базовый клиент с перехватчиками |
| **jwt-decode** | Декодирование JWT токенов для получения роли пользователя | `App.js`, `ProductsList.jsx` |
| **LocalStorage** | Хранение токенов (accessToken, refreshToken, userRole) на клиенте | `localStorage.setItem/getItem` в компонентах |
| **CSS** | Стилизация компонентов, адаптивный дизайн | `App.css` — все стили приложения |

### 🔸 Защита маршрутов

| Компонент | Назначение |
|-----------|------------|
| **PrivateRoute** | Защита маршрутов от неавторизованных пользователей. Если нет токена — редирект на логин. | `components/PrivateRoute.jsx` |
| **RoleBasedRoute** | Защита маршрутов по роли. Проверяет, имеет ли пользователь доступ к странице. | `components/RoleBasedRoute.jsx` |
| **authMiddleware** (бэкенд) | Проверяет валидность access-токена и актуальность роли из БД | `server.js` |
| **roleMiddleware** (бэкенд) | Ограничивает доступ к эндпоинтам по роли (user/seller/admin) | `server.js` |

### 🔸 Перехватчики Axios (Interceptors)

| Перехватчик | Назначение |
|-------------|------------|
| **Request interceptor** | Автоматически добавляет accessToken в заголовок `Authorization: Bearer <token>` к каждому запросу | `api/index.js` |
| **Response interceptor** | При получении ошибки 401 пытается обновить токен через `/api/auth/refresh`. Если не получается — редирект на логин | `api/index.js` |

## 📋 Функциональность и где реализовано

### Аутентификация

| Функция | Метод API | Фронтенд компонент |
|---------|-----------|-------------------|
| Регистрация | POST /api/auth/register | `Register.jsx` |
| Вход в систему | POST /api/auth/login | `Login.jsx` |
| Обновление токенов | POST /api/auth/refresh | `api/index.js` (перехватчик) |
| Получение текущего пользователя | GET /api/auth/me | `api/auth.js` |
| Выход из системы | — (очистка localStorage) | Кнопка "Выйти" в `ProductsList.jsx` |

### Управление товарами (CRUD)

| Функция | Метод API | Доступ | Фронтенд компонент |
|---------|-----------|--------|-------------------|
| Просмотр списка | GET /api/products | user/seller/admin | `ProductsList.jsx` |
| Просмотр деталей | GET /api/products/:id | user/seller/admin | `ProductForm.jsx` |
| Создание | POST /api/products | seller/admin | `ProductForm.jsx` |
| Редактирование | PUT /api/products/:id | seller/admin | `ProductForm.jsx` |
| Удаление | DELETE /api/products/:id | admin | `ProductsList.jsx` |

### Управление пользователями (только admin)

| Функция | Метод API | Фронтенд компонент |
|---------|-----------|-------------------|
| Список пользователей | GET /api/users | `UsersList.jsx` |
| Изменение роли | PUT /api/users/:id | `UsersList.jsx` |
| Блокировка | DELETE /api/users/:id | `UsersList.jsx` |
| Разблокировка | PATCH /api/users/:id/unblock | `UsersList.jsx` |

## 🔐 Система ролей

| Роль | Права доступа | Где проверяется |
|------|---------------|-----------------|
| **Гость** | Регистрация, вход | `App.js` — открытые маршруты |
| **Пользователь (user)** | Просмотр товаров | `roleMiddleware(['user', 'seller', 'admin'])` |
| **Продавец (seller)** | Всё, что у пользователя + создание и редактирование товаров | `roleMiddleware(['seller', 'admin'])` |
| **Администратор (admin)** | Всё, что у продавца + удаление товаров + управление пользователями | `roleMiddleware(['admin'])` |

## 🔒 Безопасность

| Механизм | Где реализован | Назначение |
|----------|---------------|------------|
| Хеширование паролей (bcrypt) | `server.js` | Пароли не хранятся в открытом виде |
| Access токен (15 мин) | `generateAccessToken()` | Короткоживущий токен для доступа |
| Refresh токен (7 дней) | `generateRefreshToken()` | Долгоживущий токен для обновления |
| Проверка роли из БД | `authMiddleware` | Токен не может иметь устаревшую роль |
| Аннулирование токенов при смене роли | `PUT /api/users/:id` | При изменении роли — все токены удаляются |
| Чёрный список refresh-токенов | `refreshTokens` (Set) | Украденный токен нельзя использовать повторно |
| CORS | `app.use(cors())` | Запросы только с доверенных источников |

## 📊 Схема работы аутентификации

```
1. Пользователь → Вход (email/password)
2. Сервер → Проверка пароля (bcrypt.compare)
3. Сервер → Генерация accessToken (15 мин) и refreshToken (7 дней)
4. Клиент → Сохраняет токены в localStorage
5. Клиент → При каждом запросе добавляет accessToken в заголовок
6. Сервер → authMiddleware проверяет токен и актуальность роли
7. При истечении accessToken → Перехватчик отправляет refreshToken на /api/auth/refresh
8. Сервер → Выдаёт новую пару токенов
9. Клиент → Повторяет исходный запрос с новым токеном
```

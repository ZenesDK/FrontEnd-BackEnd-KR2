## Практическая работа №9: Refresh-токены

### 📘 Описание работы

В предыдущей работе (№8) мы реализовали JWT-аутентификацию с выдачей **access-токена** (короткоживущего). На практике это приводит к тому, что каждые 15 минут пользователь вынужден заново вводить логин и пароль, что ухудшает пользовательский опыт.

**Refresh-токены** решают эту проблему. При входе в систему сервер выдаёт два токена:
- **accessToken** — используется для доступа к защищённым ресурсам (живёт 15 минут);
- **refreshToken** — используется только для получения новой пары токенов, когда accessToken истекает (живёт, например, 7 дней).

Refresh-токен хранится на клиенте (обычно в httpOnly cookie или в памяти) и отправляется на специальный эндпоинт `/api/auth/refresh`. Сервер проверяет его, и если он валиден, выдаёт новую пару токенов. При этом старый refresh-токен **аннулируется** (ротация токенов) — это повышает безопасность: если злоумышленник украдёт refresh-токен, он сможет использовать его только один раз.

### 🎯 Цель работы

- Реализовать выдачу **refresh-токена** вместе с access-токеном при входе.
- Создать эндпоинт **POST /api/auth/refresh**, который принимает refresh-токен, проверяет его и возвращает **новую пару** токенов.
- Организовать безопасное хранение валидных refresh-токенов на сервере (в данном проекте — в памяти, в реальных проектах — в базе данных).
- Интегрировать обновлённую схему аутентификации в уже существующее CRUD-приложение (пользователи, товары).

### 🔐 Что реализовано в коде

1. **Генерация токенов**
   - `generateAccessToken(user)` — создаёт access-токен с временем жизни `15m`.
   - `generateRefreshToken(user)` — создаёт refresh-токен с временем жизни `7d`.

2. **Хранилище refresh-токенов**
   - Переменная `refreshTokens = new Set()` хранит валидные refresh-токены. При логине токен добавляется, при обновлении или выходе удаляется.

3. **Маршруты аутентификации**
   - `POST /api/auth/register` — регистрация пользователя (email, имя, фамилия, пароль).
   - `POST /api/auth/login` — проверка учётных данных, возвращает `{ accessToken, refreshToken }`.
   - `POST /api/auth/refresh` — принимает `{ refreshToken }`, проверяет его, выполняет ротацию и возвращает новую пару токенов.
   - `GET /api/auth/me` — защищённый маршрут, возвращает информацию о текущем пользователе (требует accessToken).

4. **Защищённые маршруты для товаров**
   - Все CRUD-операции для товаров (`GET`, `POST`, `PUT`, `DELETE`) защищены middleware `authMiddleware`, который проверяет accessToken.

5. **Документация Swagger**
   - После запуска сервера документация доступна по адресу `http://localhost:3000/api-docs`. Там можно тестировать все эндпоинты, включая получение и обновление токенов.

### 🧪 Тестирование работы refresh-токена

Рекомендуется использовать **Swagger UI** (`/api-docs`) или любой HTTP-клиент (Postman, curl).

**Последовательность:**

1. **Регистрация**  
   `POST /api/auth/register` с телом:
   ```json
   {
     "email": "test@mail.ru",
     "first_name": "Иван",
     "last_name": "Иванов",
     "password": "123456"
   }
   ```

2. **Вход в систему**  
   `POST /api/auth/login` с телом:
   ```json
   {
     "email": "test@mail.ru",
     "password": "123456"
   }
   ```
   В ответе придут `accessToken` и `refreshToken`.

3. **Доступ к защищённому ресурсу**  
   Добавьте заголовок `Authorization: Bearer <accessToken>` и выполните `GET /api/products` — должен вернуться список товаров (пока пустой).

4. **Обновление токенов**  
   Дождитесь истечения access-токена (или измените время в коде для теста) и отправьте `POST /api/auth/refresh` с телом:
   ```json
   {
     "refreshToken": "<ваш refreshToken>"
   }
   ```
   В ответе получите **новую пару** токенов. Старый refresh-токен больше не действителен.

### ⚠️ Важные замечания

- В реальном проекте **refresh-токены должны храниться в базе данных** (например, в таблице `refresh_tokens` с привязкой к пользователю и временем истечения).
- Секреты `ACCESS_SECRET` и `REFRESH_SECRET` необходимо выносить в переменные окружения (файл `.env`) и **не хранить в коде**.
- Для повышения безопасности можно также хранить refresh-токены в **httpOnly cookies** (а не в localStorage), чтобы защитить их от XSS-атак.
